> 第 1 周学习笔记，重点是建立 OTel 整体心智模型。

---

## 1. 总览：OTel 一句话定义

**OpenTelemetry 是一套"产生、收集、传输可观测性数据"的开放标准与工具集**。

它不存数据，也不画图，它的工作就是把数据 **标准化** 后送给后端（Jaeger / Prometheus / Loki 等）。

## 3. 关键概念速记卡

### 3.1 三大支柱（Three Pillars / Signals）
| 信号 | 回答什么问题 | 例子 |
|------|--------------|------|
| **Traces** | 一次请求经过了哪些服务、每段耗时多少 | "/order/create 请求总耗时 320ms，其中 280ms 卡在 DB" |
| **Metrics** | 系统总体在如何运行（趋势/聚合） | "过去 5 分钟 QPS=120，P95=200ms" |
| **Logs** | 某一时刻具体发生了什么细节 | "user_id=42 触发了 NullPointerException" |

### 3.2 OTel 分层（自顶向下）
1. **Specification** — 协议与数据模型规范（最底层共识）
2. **API** — 语言无关接口（业务代码只引用 API，不绑死实现）
3. **SDK** — API 的官方实现（采样、批处理、属性注入）
4. **Instrumentation Libraries** — 给 Flask/Redis/Kafka 等开源库的"插桩适配器"
5. **OTLP** — 标准传输协议（gRPC 或 HTTP + Protobuf/JSON）
6. **Collector** — 独立的数据接收/加工/路由中间件
7. **Backends** — 后端存储（Jaeger/Prometheus/Loki/任意厂商）

### 3.3 为什么要有 Collector？
不直接把 SDK 连到 Jaeger 的 4 个理由：
1. **解耦**：应用不需要知道后端是 Jaeger 还是 Datadog，换后端只改 Collector 配置
2. **批量**：减少应用与远端后端之间的网络往返
3. **加工**：脱敏、采样、重写 attribute、按租户路由
4. **多目的地**：同一条 trace 同时发给 Jaeger 看 UI + S3 做归档

### 3.4 Collector 三件套
| 组件 | 作用 | 例子 |
|------|------|------|
| **Receiver** | 收数据 | `otlp`、`prometheus`、`filelog` |
| **Processor** | 加工数据 | `batch`、`memory_limiter`、`tail_sampling`、`transform` |
| **Exporter** | 发数据 | `otlp/jaeger`、`prometheus`、`otlphttp/loki`、`debug` |

## 4. Phase 1 的 curl 实验，到底验证了什么？

```
curl ─[OTLP/HTTP JSON]─▶ Collector(otlp receiver) ─[batch]─▶ Collector(otlp/jaeger exporter) ─[gRPC]─▶ Jaeger ─▶ UI
```

✅ 验证了 **左半边业务无关 → 右半边 UI 显示** 之间的所有"管道"都是通的。

❌ 还没用到的部分：
- Prometheus / Loki / Grafana（要等 Phase 3 / 4 发指标和日志时才会激活）
- SDK / 自动插桩 / 真实业务（Phase 2 起开始用 Python 代码）
